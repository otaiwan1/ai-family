# AI Family Feud - 開發計畫 (Project Plan)

## 階段一：AI 核心機制概念驗證與「題庫建立」 (PoC & Database Prep)
**目標：驗證連續詢問 AI 相同問題並統計答案的可行性，並將結果存為遊戲題庫。**
因為即時詢問 100 次會太慢，我們必須採用「預先生成 (Pre-generation)」的策略。
- [ ] 準備一份種子問題清單（例如 10~20 個有趣的開放式問題）。
- [ ] 建立基礎 Node.js 或 Python 腳本串接 OpenRouter API (`google/gemma-4-31b-it:free`)。
- [ ] 設定 prompt 與模型參數（如 `temperature`），測試同個問題連續詢問 50-100 次。
- [ ] **實作「答案正規化（Normalization）」機制**：如何透過腳本或再呼叫一次 AI 來將相似/同義答案合併統計（例如「蘋果」、「Apple」、「一顆蘋果」）。
- [ ] 將最終處理好的各題排行榜，自動產出並存檔成一個靜態的 JSON 檔案（例如 `questions_db.json`），作為後續遊戲的資料庫。

## 階段二：雙視窗網頁系統開發 (Host & Audience Views)
**目標：建立主持人控制台與觀眾大螢幕，並透過 WebSocket 保持兩畫面同步。**
使用「網頁實作」是最佳方案，這讓主持人可以用平板或筆電獨立控制，並將另一個視窗全螢幕投影給觀眾看。
- [ ] 選擇前端框架（推薦 React/Vite + Tailwind CSS 開發畫面）。
- [ ] 建立後端 WebSocket 伺服器：負責管理目前在哪一題、翻開了哪些答案、分數與 ❌ 的狀態，並即時廣播給 Host 與 Audience 視窗。
- [ ] 實作**觀眾畫面 (Audience View)**：
  - 刻畫經典的「翻牌大螢幕」UI：隱藏狀態的答案板、排名、計分板、3 個 ❌ 的出現動畫。
  - 實作答對音效（叮咚）與答錯動畫（大大的紅色 ❌ 配上蜂鳴器聲音）。
- [ ] 實作**主持人畫面 (Host View)**：
  - 顯示當前題目的所有解答、答題人數與排名（僅主持人可見）。
  - 控制按鈕：切換下一題、點擊翻開某張牌、按下 ❌ 給予警告。

## 階段三：串接題庫與過場動畫優化 (Data Integration)
**目標：正式載入 `questions_db.json`，並優化遊戲環節間的體驗**
- [ ] 寫入腳本或 API，讓前端能確實讀取 AI 生成的排行榜題目。
- [ ] 補上回合與回合之間的結算邏輯（搶分階段、把分數結算給 Team A 還是 Team B 等）。
- [ ] **實作分數匯出 (Export) 與重置 (Reset) 功能**，方便活動賽後統計與重新開始新的一局。

## 階段四：進階功能與多人連線 (可選/未來擴充)
- [ ] 實作 WebSockets，讓玩家可以用手機當遙控器/搶答器。
- [ ] 讓玩家可以自己輸入問題，系統即時平行呼叫 AI 生成排行榜（需要處理等待時間的過場動畫）。
